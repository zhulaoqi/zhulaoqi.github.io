RAG已死？LangChain过时？一个高级工程师的冷静复盘

发表于 2026-03-19 分类于 技术思考
本文字数： 5.7k 阅读时长 ≈ 5 分钟

2024 年底到 2026 年初，AI 工程领域经历了一场剧烈的范式震荡。Gemini 把上下文窗口推到了 1M+ tokens，Anthropic 发布了 MCP（Model Context Protocol），OpenAI 的 Function Calling 成为事实标准，Cursor/Windsurf 等 AI IDE 用 Skills + Prompt 的方式重新定义了工程效率。

于是那两个曾经的”当红炸子鸡”被推上了审判席：RAG 是不是已经消亡了？LangChain 是不是彻底过时了？

作为一个在生产环境中踩过这两个坑、也用它们交付过项目的工程师，我想给一个不那么情绪化的答案。

一、先说结论

技术 判决 一句话总结
Naive RAG ☠️ 已死 把文档切片扔向量库然后祈祷的时代结束了
Advanced RAG ✅ 活着且在进化 GraphRAG、Agentic RAG、Hybrid Search 正当壮年
LangChain ⚠️ 重度过时 90% 的场景不再需要它，但 LangGraph 还有一席之地
MCP 🚀 新基建 正在成为 LLM 工具调用的 USB-C 接口
Prompt/Skills 🔥 主流范式 Context Engineering 才是 2026 年的核心能力

二、RAG：不是死了，是脱胎换骨了

2.1 Naive RAG 的死因

2023 年的标准 RAG pipeline 长这样：

这个 pipeline 在生产中暴露了三个致命问题：

第一，切片即失忆。 一份 50 页的技术方案被切成 100 个 chunk 后，跨章节的逻辑关系全部丢失。你问”方案的整体架构是什么”，检索回来的是三段不相干的文字碎片。

第二，Top-K 是赌博。 语义相似度 ≠ 问题相关度。用户问”系统的瓶颈在哪”，召回的可能是包含”瓶颈”这个词但讨论完全不同主题的段落。

第三，长上下文窗口来了。 当 Gemini 1.5 Pro 能吃下 1M tokens 的时候，很多人的第一反应是——还要什么 RAG？直接把整个文档塞进去不就完了？

2.2 但长上下文并不是银弹

先看一组实测数据：

指标 RAG (检索+生成) 长上下文 (全量灌入)
平均延迟 ~1s ~45s
单次成本(100K tokens 文档) ~$0.01 ~$0.30
“Lost in the Middle” 准确率衰减 无（精准定位） 30%+ 衰减
动态知识更新 实时 需重新加载全量
可溯源性 天然支持引用来源 几乎不可能

关键发现： 前沿模型在超过 32K-64K tokens 后，注意力开始严重衰减。所谓的 1M 上下文窗口，在”大海捞针”测试中表现优秀，但在真实的多跳推理场景中远不如预期。

长上下文窗口解决的是**”我有一份完整文档想让 AI 理解”的问题。RAG 解决的是“我有 10TB 的知识库，需要秒级精准定位”**的问题。两者解决的根本不是同一个问题。

2.3 2026 年 RAG 的真实形态

如果你还在用 2023 年的 Naive RAG 架构，那确实该淘汰了。现代 RAG 已经进化成了这样：

核心变化：

  • Hybrid Search：向量检索 + BM25 + 知识图谱的混合检索，互相补位
  • Agentic 循环：检索结果不满意？自动改写 query 重新来
  • Reranker：Cross-Encoder 对初筛结果精排，干掉”语义相似但不相关”的噪声
  • GraphRAG：用知识图谱保留文档间的逻辑关系，解决切片失忆问题

我的判断：RAG 没有死，Naive RAG 死了。现代 RAG 是一个检索工程+LLM 编排的系统工程，不是一个 pip install 就能搞定的玩具。

三、LangChain：从”万能框架”到”过度工程”

3.1 LangChain 的问题出在哪

我在 2023 年用 LangChain 做过两个项目。说实话，第一个月很爽——开箱即用，chain 一串就跑。第三个月开始痛苦——调试像噩梦，抽象层层嵌套，一个简单的 API 调用被包了五层。

LangChain 的核心问题：

过度抽象。 为了追求通用性，LangChain 把每个概念都抽象了一层又一层。你想调用一个 OpenAI API，需要经过 ChatOpenAI → BaseChatModel → BaseLanguageModel → Runnable 四层继承。调试时 stack trace 长到怀疑人生。

实测对比——用 MCP 和 LangChain 实现同一个功能：

MCP 方案代码量是 LangChain 的 1/4，开发效率快 3 倍，而且不绑定任何特定框架。

3.2 LangChain 还有什么价值

说句公道话，LangChain 并非一无是处：

  • LangGraph：专注于有状态的多步骤 Agent 工作流编排，这一块确实做得不错
  • LangSmith：LLM 应用的可观测性平台，在生产监控方面有独到价值
  • 教学价值：对于理解 LLM 应用架构的概念模型仍有参考意义

但核心的 Chain/Agent 抽象层，在 2026 年的生产环境中已经是负资产。

四、2026 年的技术棋盘：谁在取代谁

4.1 全景对比

维度 RAG LangChain MCP Function Calling Prompt/Skills
本质 知识检索增强 应用开发框架 工具调用协议 API 原生能力 上下文工程
解决什么 LLM 不知道的知识 胶水代码 工具互联互通 单次工具调用 引导 LLM 行为
复杂度 中-高 高 低 极低 低-中
厂商锁定 无 轻度 无 重度 无
生产就绪 ✅ ⚠️ ✅ ✅ ✅
2026 趋势 进化中 萎缩中 快速扩张 稳定 主流化

4.2 MCP：LLM 世界的 USB-C

MCP 之所以重要，是因为它解决了一个之前被忽视的架构问题：每个 AI 应用都在重新发明轮子来连接外部工具。

在 MCP 之前，你要让 LLM 读数据库、查日历、操作文件，每个集成都要自己写 adapter。LangChain 试图用”Toolkit”抽象解决这个问题，但本质上是把一堆 adapter 打包进一个框架。

MCP 的做法是定义协议而非框架：

  • Server 暴露工具和资源（标准化接口）
  • Client 自动发现和调用（运行时发现，不是编译时绑定）
  • 与具体 LLM 无关（Claude、GPT、Gemini 都能用）

这就像 USB-C 之于各种充电线——不是又造了一根线，而是让所有线都用同一个口。

4.3 Prompt Engineering → Context Engineering

2026 年最重要的认知升级是：我们谈论的不再是”Prompt Engineering”，而是”Context Engineering”。

五、给工程师的务实建议

5.1 现在该学什么

  1. Context Engineering：这是 2026 年 AI 工程师的核心技能。理解如何设计 System Prompt、如何编排 Tool Definitions、如何管理上下文窗口
  2. MCP 协议：学会写 MCP Server，这是未来工具集成的标准方式
  3. 现代检索工程：如果你要做 RAG，学 Hybrid Search + Reranker + Knowledge Graph，别停留在 naive vector search
  4. 直接调 SDK：大多数场景直接用 OpenAI/Anthropic SDK + Function Calling 就够了，不需要框架

5.2 现在该扔什么

  1. 扔掉 Naive RAG pipeline——切片+向量搜索+Top-K 的原始方案
  2. 扔掉 LangChain 的 Chain/Agent 抽象——直接用 SDK 或 MCP
  3. 扔掉”一个框架解决所有问题”的思维——组合轻量工具比全家桶更灵活
  4. 扔掉”Prompt Engineering 就是写 prompt”的认知——升级到 Context Engineering

5.3 别犯的错误

  • 别因为有了长上下文就不做 RAG：10TB 的企业知识库不可能全塞进去
  • 别因为 LangChain 过时就否定框架的价值：LangGraph 在复杂工作流编排中仍然有用
  • 别追新概念忘了基本功：好的检索工程、好的 prompt 设计、好的系统架构，这些永远是基本盘

六、最后

技术世界没有”已死”或”永生”，只有适配度。RAG 在它擅长的场景依然不可替代，LangChain 在大多数场景确实过度工程了，MCP 和 Context Engineering 正在成为新的基础设施。

作为工程师，最危险的不是用了”过时”的技术，而是用了”不匹配”的技术。在你的场景里，什么方案能用最少的复杂度解决问题，什么就是最好的方案。

不要信仰技术，要信仰 trade-off。

本文基于 2026 年 3 月的技术生态撰写，AI 领域变化极快，半年后可能又是另一番景象。保持学习，保持怀疑。
